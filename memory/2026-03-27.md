
## Medication Photo Rule (NON-NEGOTIABLE)
- Photos submitted to the Food Log group that show pills/medication are VALIDATION that the dose was taken — NOT a new/additional dose to log.
- When a meal photo is processed and medication is detected or auto-logged at a nearby timestamp, do NOT create a second medication entry. The auto-tracker (auto_track_meds.js) handles the scheduled dose; the photo is confirmation only.
- If two Metformin (or any med) entries exist for the same meal window (e.g., 13:00 and 13:36 both labeled "lunch"), the duplicate is the auto-logged one — keep the entry tied to the actual meal time and remove the scheduled one.

## Fix Discipline Rules (NON-NEGOTIABLE — from Javi 2026-03-28)
1. **No fix goes in without a test** — test in the ACTUAL execution environment, not just the shell. For cron scripts: simulate with restricted PATH (`env -i PATH=/usr/bin:/bin node script.js`) before declaring fixed.
2. **Verify data state before declaring success** — not just the immediate write, but what the next cron run will do. Check what crons are scheduled to run next and whether they will undo or corrupt the fix.
3. **Stop the bleeding before patching** — when a bug is actively corrupting data (e.g. wrong timestamps, duplicate entries), pause the relevant crons FIRST, then fix, then re-enable. Never let broken crons keep running during a repair.

These rules apply to ALL fixes, no exceptions. Claiming "fixed" without following all 3 is not acceptable.
