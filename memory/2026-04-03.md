# April 3, 2026 — Photo URL Pipeline Rebuilt

## Root Cause Analysis Complete
**Why photos never got URLs attached:**

The pipeline chain was built on a **dead source**:
- `telegram_ingest_updates.js` uses Telegram's `getUpdates` **polling API**
- OpenClaw switched to **webhook delivery** sometime in late March
- Since then, `getUpdates` returns nothing (webhook consumed all updates)
- Last update fetched: March 28, 02:07 UTC
- Photos from March 29 onwards: envelope metadata **never written** to `telegram_media_envelopes.jsonl`
- Without envelopes, `photo_to_log_pipeline.js` and `auto-fix-pending-photo-refs.js` had no caption/timestamp/metadata to link the photo to the log entry
- Result: **100% of photos from March 29–April 3 silently skipped**

The scripts were working correctly within their own logic. They were just looking at empty data.

## Solution: Move Responsibility to HealthGuard
**Who should own photo upload:** The agent that actually receives the photo webhook (HealthGuard), not a background polling job.

**Changed:**
1. Updated `health-guard.md` with real-time photo upload directive:
   - When Maria sends a food photo, HealthGuard immediately uploads to freeimage.host
   - Appends `[photo](URL)` to health_log.md entry **in the same turn**
   - No deferral, no envelope bridge, no polling

2. Disabled dead polling jobs:
   - `Photo Pipeline - Process Inbound Photos` (every 1m → disabled)
   - `auto-fix-pending-photo-refs` (every 30m → disabled)

3. Why not keep them as fallback?
   - They're 1440 + 48 = 1488 runs/day doing **nothing** (0% yield rate)
   - They were never the real problem (envelope source is broken)
   - Keeping them creates false impression of redundancy where there is none

# April 3, 2026 (Evening) — Cron/Infrastructure Engineering + Cost Analysis

## System Cron PATH Fix (Critical)
**Root cause of backups.json staleness alerts:** System cron runs with `PATH=/usr/bin:/bin` — no Homebrew. Scripts that internally called bare `execSync('node ...')` were silently failing with "command not found". Fix: patched all scripts to use full path `/opt/homebrew/bin/node` in any internal `execSync` calls. This was the real root cause of persistent stale backup alerts, not the pagination loop (which was a separate issue).

## MySQL Glucose Sync Pagination Fix
- Added 48-hour age threshold cutoff in `mysql_glucose_sync.js`
- Stops pagination when oldest batch entry >48h old
- Root cause: `INSERT IGNORE` skips duplicates silently, loop never knew it was caught up
- Committed to git with detailed explanation

## Cron Architecture
- **System crontab** (pure scripts, no agent overhead): Radial Sync (5,35), MySQL Glucose (20,50), Notion Impact (0 * * * *), Glucose Low Alert (*/5), MySQL Daily Backup (20 4 * * *), Notion→MySQL Sync (0 */4 * * *)
- **OpenClaw cron** (LLM-dependent only, 5 jobs): daily-log-review, maria-glucose-summary, health-sync-daily-audit, weekly-memory-summary, cron-health-watchdog

## API Cost Analysis (Anthropic, post-switch ~10 days)
- Javi's subscription: **$100/month**
- Normal day API cost: ~$2.68/day → **$80/month**
- Dev/engineering day: $15–40/day
- Estimated monthly API bill (mixed): **$150–220/month** (above subscription cost)
- Conclusion: Subscription is likely better value unless dev days are rare and light
- Key caveat: Only ~10 days of Anthropic-as-primary usage; sample is biased toward heavy dev days; need 2–3 more weeks of normal usage data for accurate estimate
- Anthropic is offering one-time credit + discounted bundles as part of OpenClaw subscription sunset (effective April 4, 2026, 12pm PT)

## Prediction Model v3 (Finalized in AGENTS.md)
- 4-layer formula: carb factors by range (2.0/1.3/1.2/0.8) + meal intercepts (Breakfast+31, Lunch-12, Dinner-2, Snack+4, Dessert-14) + cumulative preBG anchor + time-to-peak defaults
- Expected accuracy: ~87–89% within ±20 mg/dL
- Cumulative meal rule: food within 2h = same meal type, use cumulative carb sum, anchor to first item's preBG

- Options: Anthropic API key (pay-as-you-go) or switch primary model to free alternatives (kimi, Codex)

## Claude Code Sandbox Issue — Confirmed & Understood
**Problem:** Claude Code running directly from Mac terminal (`cd ~/.openclaw/workspace && claude`) returns "0 crons" for both `crontab -l` and OpenClaw CronList, despite 7 system cron entries and 5 OpenClaw jobs existing.

**Hypothesis 1 (wrong):** Claude Code is spawned through OpenClaw ACP → runs in Docker container → sandboxed. Javi confirmed he runs Claude Code directly from Terminal, not through OpenClaw.

**Hypothesis 2 (under investigation):** Claude Code may be running bash in its own managed subprocess environment that doesn't inherit the correct user environment, OR it's using a different user context than expected, OR it's parsing `crontab -l` output incorrectly (filtering comments and returning 0).

**Next step:** Need Javi to run `crontab -l | wc -l` directly in Claude Code and report back. If it returns the correct number (7+), the issue is Claude Code's output parsing, not actual sandbox isolation.

**Workaround created:** `scripts/dump_system_state.sh` — exports crontab + OpenClaw cron list + sync state to `data/system_state.md` for Claude Code to read as a static file. Note: this script must be run from a real terminal (not inside Claude Code) to be useful.

## Key Insight
If Claude Code truly can't see crontab entries from a direct terminal launch (not ACP), the explanation isn't sandboxing — it may be a Claude Code bug where it wraps bash differently on macOS, or the cron tool it's calling isn't `crontab -l` at all but OpenClaw's API which requires auth it doesn't have in session.
